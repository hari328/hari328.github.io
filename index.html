Learning DevOps as a Developer

March 19, 20263 min read

If someone asks you to build an app, they're really asking: can you build it, ship it, keep it running, and help it grow? Most developers are comfortable with the first part. The rest is where things get fuzzy.

This series is about closing that gap.

The Business Owner's View

When a business owner thinks about their product, they're thinking about:

  1. Deployment — the product they use
  2. Monitoring — how is my product doing, where can I see when something goes wrong
  3. Cost — how much does running this cost me

Code? They don't care about that. But everything above depends on you knowing how to ship and run it.

The Path

  1. CI/CD with GitHub Actions — making sure code is good before it ships, and packaging it for the cloud
  2. Terraform + Terragrunt — defining your infrastructure as code
  3. AWS — VPC, subnets, ECS, ECR, IAM and everything that actually runs your app
  4. Terraform + Terragrunt + GitHub Actions — the full picture: infrastructure meets deployment pipeline
  5. Monitoring basics — USE and RED, the mental models for understanding system health
  6. Observability tooling — traces, logs, metrics, and Alloy
  7. Running everything locally — the full stack on your machine

The Code

All examples in this series are drawn from two repos:

  • backend-patterns-ts — the application: two TypeScript microservices communicating via AWS SQS
  • infra-backend-patterns-ts — the infrastructure: Terraform/Terragrunt config for everything on AWS

Deployment: CI/CD with GitHub Actions

March 19, 202612 min read

CI vs CD

Deployment is really two things:

CI (Continuous Integration) — making sure that whatever code is being merged into the codebase is good enough. Tests pass, types check out, nothing is broken.

CD (Continuous Deployment/Delivery) — having a clean, repeatable pipeline to ship code to production whenever you're ready. When you deploy depends on the project and company, but having the pipeline ready means you're never scrambling.

The easiest way to learn both is GitHub Actions.

What is GitHub Actions?

Think of GitHub Actions as a pipeline. Some event — a pull request, a push to main, a manual trigger — causes a series of automated steps to run in response. You define what those steps are in a YAML file that lives inside your repo under .github/workflows/.

PR opened → CI runs tests → merge → build Docker image → deploy to cloud

That's the full loop. Let's walk through each part using a real project: backend-patterns-ts, a TypeScript monorepo with two microservices communicating via AWS SQS.

Anatomy of a Workflow

Every GitHub Actions workflow has the same building blocks.

1. Trigger

The on key defines what kicks off the workflow.

on:
  pull_request:
    branches: [main]

This runs whenever a PR targets main. You can trigger on push, on a schedule, or manually.

2. Jobs and Runners

A workflow is made up of jobs. By default, jobs run in parallel. Each job runs on a runner — a VM that GitHub spins up for you.

jobs:
  lint:
    runs-on: ubuntu-latest
  test:
    runs-on: ubuntu-latest

Both jobs start at the same time. If you want one to wait for another, you use needs.

3. Steps

Each job has steps that run sequentially, top to bottom. A step either runs a shell command or calls an action.

4. Actions

Actions are reusable steps that the community (or GitHub itself) has published. Instead of writing 20 lines to checkout code or configure AWS credentials, you call a single action.

- uses: actions/checkout@v4
- uses: actions/setup-node@v4
  with:
    node-version: 22

5. Services

Services are containers that run alongside your job — things like a database, cache, or cloud emulator that your tests depend on. Instead of mocking these dependencies, you run the real thing.

services:
  postgres:
    image: postgres:16-alpine
    ports:
      - 5432:5432

  localstack:
    image: localstack/localstack
    ports:
      - 4566:4566

  redis:
    image: redis:7-alpine
    ports:
      - 6379:6379

The real power here is LocalStack — it emulates AWS services like SQS locally inside the runner. Your integration tests send messages to a fake SQS queue on localhost:4566 without ever touching real AWS. No costs, no credentials, no flaky network calls.

6. Environments and Secrets

Environments let you add protection rules and approvals to sensitive workflows. Secrets store sensitive values like API keys or connection strings, injected at runtime as ${{ secrets.DATABASE_URL }}.

7. OIDC

OIDC (OpenID Connect) lets your workflow get temporary, short-lived credentials from cloud providers like AWS — without storing long-lived keys as secrets. GitHub and AWS establish trust via a token that expires when the job finishes. This is the right way to authenticate to the cloud from GitHub Actions.

Act 1 — CI: Does My Code Work?

The first pipeline you should build is CI. Run it on every pull request so you catch problems before they land on main.

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

Parallel jobs for fast feedback

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm ci
      - run: npm run lint

  type-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm ci
      - run: npm run type-check

Lint and type-check run at the same time. No waiting. If either fails, the PR is blocked.

Services: real dependencies for integration tests

Integration tests need real infrastructure, not mocks. GitHub Actions lets you run Postgres, Redis, and LocalStack as sidecar containers right inside the runner. This repo uses AWS SQS to pass messages between microservices. In tests, LocalStack intercepts those SQS calls so you're testing the real message flow without hitting real AWS.

Controlling job order with needs

  unit-tests:
    needs: [lint, type-check]

  integration-tests:
    needs: [lint, type-check]

The needs key creates a dependency. Unit tests and integration tests only start after lint and type-check both pass. No point running tests if the code doesn't even compile.

Act 2 — Build: Package It for the Cloud

Once code lands on main, the next step is packaging it as a Docker image and pushing it to a registry.

on:
  push:
    branches: [main]
    paths:
      - 'apps/**'
      - 'packages/**'

This only triggers when app code changes — not when docs or config files are updated.

Tagging images with the git SHA

Every image needs a unique tag. Using the git commit SHA means you can always trace a running container back to the exact code it was built from:

- name: Set image tag
  id: meta
  run: |
    SHORT_SHA=$(git rev-parse --short HEAD)
    echo "image_tag=${SHORT_SHA}" >> "$GITHUB_OUTPUT"

OIDC: no long-lived AWS keys

Most tutorials tell you to create an IAM user, generate an access key, and paste it into secrets. Don't do this. Keys get leaked. Keys need rotating. Keys are a liability.

Use OIDC instead. GitHub gets a temporary token from AWS that expires when the job finishes:

permissions:
  id-token: write
  contents: read

- name: Configure AWS Credentials via OIDC
  uses: aws-actions/configure-aws-credentials@v4
  with:
    role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
    aws-region: us-east-1

The only secret you need is AWS_ROLE_ARN — just an ARN string, not a credential. AWS verifies the token, grants temporary access to the role, and the job proceeds.

Building and pushing to ECR

- name: Login to Amazon ECR
  id: ecr-login
  uses: aws-actions/amazon-ecr-login@v2

- name: Build & push posts-service
  run: |
    IMAGE="${{ steps.ecr-login.outputs.registry }}/${PROJECT}-posts-service"
    docker build -f apps/posts-service/Dockerfile \
      -t "${IMAGE}:${IMAGE_TAG}" \
      -t "${IMAGE}:latest" .
    docker push "${IMAGE}:${IMAGE_TAG}"
    docker push "${IMAGE}:latest"

Two tags per image: the specific SHA (immutable, used for deploys) and latest (for convenience).

The Full Picture

PR opened
  └── ci.yml → lint, type-check, unit tests, integration tests
               (with real Postgres, SQS via LocalStack, Redis)

Merged to main
  └── build-push.yml → build Docker images
                     → tag with git SHA
                     → push to ECR (via OIDC, no long-lived keys)

Two workflows. Two concerns. Each small enough to understand on its own.

At this point you have images built and sitting in ECR. The next question is: where do they actually run? That requires understanding AWS — ECS, networking, IAM roles. That's covered in the next article.
